# Phase 0 Observations

## 我們做出了什麼？

- 在 `src/app/App.tsx` 入口維持兩個分頁：「原始資訊」與「整理工作台」。
- 工作台為 12 筆 mock 原始資訊預設建立保守的安全草稿（candidate type 為 unknown、confidence 為 low、unsafe 為 true）。
- 工作台支援編輯草稿：可修改候選類型、信心程度、下一步、判斷依據、卡住點與人工備註。
- 原始資訊與工作台都顯示自動資料品質提示：第三方轉述、地點可能不精確、時間或狀態不確定。這些提示只是 heuristics，僅供人工參考。
- 右側完成檢查會即時統計草稿數量、不宜直接行動數量、有卡住點數量、第三方轉述數量與人工修正數量。
- 工作台新增「人事時地物檢查表單」：只有填完人、事、時、地、物、聯絡等必填欄位，才能送出「候選通報」。送出後仍標示為未確認的候選狀態，不會變成正式任務或真的派出志工。
- 表單支援「緊急」模式：勾選後只需要填寫地點、簡略事件、受害者情形說明，就能送出「緊急候選通報」。緊急通報在畫面上會以紅色邊框與橘色狀態標籤明顯標示，但仍然不是已確認事實。
- 表單新增「嚴重性」欄位：緊急模式預設為「重（紅）」，一般模式可選「輕（綠）」或「中（黃）」。嚴重性只影響卡片顏色排序，不影響查核狀態。
- 首頁新增第三個分頁「已通過檢查」，以卡片形式列出所有候選通報，並依嚴重性（紅→黃→綠）排序。
- 候選通報支援「編輯」與「撤銷」：在工作台的已建立通報卡片上可以點「編輯通報」回到表單修改；也可以點「撤銷通報」把它刪除。撤銷後可以重新建立新的派遣建議。

## 哪些資料讓 agent 很容易整理？

- M-009 與 M-010 的時間、地點、條件與限制相對清楚，雖然仍標為 `needs_review`，但已經可以嘗試整理成候選結果。
- M-002 雖然是物資狀態，但「下午還有沒有」讓它不能直接使用。

## 哪些資料讓 agent 亂補或誤判？

- 如果要求 agent 直接幫每筆資料「分類」，agent 可能會把 M-001 的「老雜貨店後面」補成假設地址，或把 M-005 的「官方公告」sourceType 誤認為已查核。
- 我們因此只保留保守預設，所有 candidate type 初始都設為 unknown，強迫人類在畫面中判斷。

## 哪些資訊不能直接變成任務？

- M-001：地點只有「老雜貨店後面」，人數「十幾個」不精確，且是「有人說」的二手資訊。
- M-005：截圖沒有日期、沒有官方查證、沒有確切路段。
- M-008：「A 區先不要再派人」但沒有說明原因，無法決定下一步。
- M-011：志工代長者轉述，地點模糊且尚未確認長者是否同意公開地址。
- M-012：家屬來電，不在現場，也無法確認親友位置。

即使有「人事時地物」檢查表單，原始資料仍必須先被確認。表單只是強迫補齊資訊，不表示送出後就變成已查核任務。

## 哪些資訊的操作者不是當事人？

- M-001：「有人說」需要清泥，回報者未必是需求方。
- M-011：現場志工代長者轉述。
- M-012：外地家屬來電，家屬不是受困者本人。

## 我們剛剛偷偷假設了什麼？

- 我們假設「出現某些關鍵字」就代表第三方轉述、地點模糊或時間不確定。這只是輔助閱讀，不是真正語意理解。
- 我們假設 `verificationStatus` 是 `needs_review` / `unverified` 的資料都不宜直接行動。這是第一階段的安全預設，後續階段可能會有不同的狀態機。
- 我們假設「勾選緊急」就可以降低必填欄位。這是畫面上的妥協，不代表緊急通報比較可信；它只是讓協作者更快看到最低限度資訊，再補齊缺口。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 需要明確的資料格式（schema）：例如地點必須有「可導航地址」欄位、任務必須有「當事人聯繫方式」欄位。
- 需要人工確認角色：誰負責查證？誰能標記 verified？誰能把候選資訊轉成正式任務？
- 需要時間有效性的處理：資訊在幾分鐘後過期、誰負責更新？

## 我們需要更清楚資料格式的地方

- 目前 `messy-reports.json` 只有 `id`、`rawText`、`sourceType`、`verificationStatus`、`updatedAt`，沒有地點、聯絡人、當事人、有效期限等欄位，導致無法直接整理成任務。
- 我們在工作台用表單強迫補齊「人事時地物 + 聯絡」，但這是畫面層的檢查，不是資料格式的保證。後續若要自動化，需要把這些欄位寫進資料 schema。
- 「嚴重性」目前只影響卡片顏色與排序，尚未和查核狀態、資源調度規則連動。後續如果想要根據嚴重性自動提醒，需要明確的狀態機與人工授權規則。
- 後續如果要承接這些候選資訊，需要區分「來源」、「整理者」、「查證者」、「執行者」四種角色。

## 我們需要人工確認的地方

- 所有 `verificationStatus` 為 `needs_review` / `unverified` 的資料都需要人工確認，目前沒有任何一筆可直接當成已確認事實。
- M-009 雖然相對完整，但它寫「尚未看到官方公告同步更新」，因此仍需人工確認是否等同官方集合點。
- M-011 必須先確認長者是否同意公開完整地址與接受協助，才能決定是否建立任務。
